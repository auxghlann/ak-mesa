*This is a software artifact from my thesis. Our ITE department praises it for it's potential, and there is now an ongoing talk that they might adapt it on their current curriculum.*

## The Problem Statement
Abstract (taken from my thesis): 

*The use of Generative Artificial Intelligence tools in education is becoming increasingly apparent each year due to the benefits they offer. It can provide fast, easy access to information; however, such accessibility may hinder the learning process and introduce risks, such as dependency, reduced critical thinking, and concerns about academic integrity. Such risks are evident with programmers who are still learning fundamental skills. Their early contact with AI tools led to reliance on generated output rather than encouraging independent problem-solving, weakening long-term understanding. With AI capable of producing near-accurate outputs with such ease, it is becoming hard for instructors to know whether such work indeed reflects genuine learning. As a result, assessment methods based on outputs are becoming insufficient, showing the need for process-oriented assessment and a new definition of AI’s role in learning. This study addressed these challenges through rbAI, a coding playground integrated with engagement monitoring and a guided AI assistant. It introduced an engagement model that monitors key interaction patterns to evaluate the student engagement while reframing AI as a thinking partner that guides learners through problem-solving without promoting cognitive shortcuts.*

## Architecture & Tech Stack
To handle real-time telemetry processing alongside autonomous AI planning and safe code execution, the system is decoupled into three core services orchestrated via **Docker Compose**:
1. **Frontend Client**: Built with `React, TypeScript, and Vite`. It utilizes a `Monaco Editor` for the coding playground and captures high-resolution telemetry events (focus changes, keypress intervals, run attempts).
2. **Backend API & Data Engine**: A `FastAPI` application in `Python` backed by `SQLite` (for prototype) and `PostgreSQL` (in production). This handles session coordination, data fusion for the behavior engine, and hosts the `LangGraph` chat runtime.
3. **Execution Service**: A completely isolated microservice (using `docker`) specifically for running student code submissions safely against validation test cases.

## The Design
The most complex technical hurdle was bridging the gap between raw UI telemetry and meaningful pedagogical intervention. I needed to quantify "learning" versus "slacking" or "cheating."
### 1. The Behavior-Driven Engagement Model
To translate behavior into actionable data, I built a data fusion engine that calculates an **Engagement Score**. This pipeline filters out noise (like spamming random keys to inflate metrics) and distinguishes between actual off-task behavior and productive, reflective pausing.

### 2. Socratic Pedagogy
Even with perfect behavioral insights, standard LLMs tend to be overly helpful. 
With that, I implemented a strict **Pedagogical Firewall** inside the **LangGraph** runtime. 
Before any prompt reaches the core AI, the system analyzes the student's request to select an appropriate response. This allows the system to refuse to provide direct answers or code solutions, and only allow guided response to the students 
to encourage them to solve the problem themselves without spoonfeeding them the solution.

Through this workflow, the system's AI assistant guides the student with their task at hand augmenting their human capabilities. In this way,
the system is not promoting overreliance and total replacement to the students cognitive load. 


## Lessons Learned
Building rbAI taught me the intricate realities of applying AI to human-centric problems. I learned how to translate highly subjective states (learning, struggling, reflecting) into resilient, deterministic algorithmic pipelines that can't be easily gamed.
Furthermore, integrating LangGraph completely shifted my perspective on modern LLM development. Moving away from standard conversational endpoints, I learned how to build true autonomous agents with explicit state tracking, bounded tool execution, and complex routing architectures. Ultimately, I proved to myself that sometimes the best way to leverage AI isn't to build a tool that does the work for you, but one that effectively challenges you to do it yourself.
